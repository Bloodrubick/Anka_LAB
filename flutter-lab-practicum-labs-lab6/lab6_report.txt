ОТЧЕТ ПО ЛАБОРАТОРНОЙ РАБОТЕ №6
Тема: Автоматизированное тестирование во Flutter (Unit, Widget, Golden тесты)

1. ЧТО БЫЛО СДЕЛАНО:
Целью работы была реализация полного набора автоматизированных тестов для приложения, чтобы обеспечить покрытие бизнес-логики, пользовательского интерфейса и визуальной верстки. Работа выполнена на продвинутом уровне (оценка "4").
Были написаны и успешно пройдены:
- Unit-тесты для репозиториев (PlacesRepository, SearchRepository), конвертеров (FavoritePlaceFromDbConverter, FavoritePlaceToDbConverter) и бизнес-моделей (SettingsModel).
- Widget-тесты для проверки функционала и наличия элементов на экране (FavoritesButtonWidget, BottomNavScreen).
- Golden-тесты для попиксельной проверки отображения карточек (PlaceCardWidget, SkeletonPlaceCardWidget).

2. КАК БЫЛО РЕАЛИЗОВАНО:
- В качестве основных инструментов использовались встроенный пакет `flutter_test` и пакет `mockito` для генерации заглушек (моков).
- Для тестирования репозиториев с помощью аннотации `@GenerateMocks` были созданы моки для `ApiClient`, `IFilterStorage`, `HistorySearchDao` и других зависимостей. 
- Настроены проверки успешных ответов от сервера (ResultOk) и правильной обработки ошибок (ResultFailed).
- Для тестирования виджетов создавались тестовые обертки (`makeTestableWidget`) на базе `MaterialApp`, куда прокидывались замоканные зависимости и стейт-менеджеры (например, `FakeAppScope`).
- С помощью команды `flutter test --update-goldens` были сгенерированы эталонные скриншоты для Golden-тестов, с которыми в дальнейшем сверяется верстка.

3. С КАКИМИ ТРУДНОСТЯМИ СТОЛКНУЛИСЬ И КАК ИХ РЕШИЛИ:

Трудность А: Зависание и падение Golden-тестов с ошибкой `MissingPluginException` (отсутствие `path_provider` в тестовой среде).
Решение: Ошибка возникала из-за компонента `CachedNetworkImage`, который пытался обратиться к локальному кэшу при наличии пустого URL в тестовых данных. В тестовых DTO-моделях был передан пустой список картинок `images: []` вместо `['']`, что полностью предотвратило сетевые запросы при тестировании верстки.

Трудность Б: Падение виджет-теста `BottomNavScreen` с ошибкой `MissingStubError` при сборке дочерних экранов.
Решение: Экраны внутри `AutoTabsScaffold` требовали множество зависимостей (API клиент, БД, настройки) из `IAppScope`. Использование автосгенерированного `MockIAppScope` приводило к ошибкам из-за несовпадения типов и отсутствия заглушек. Был написан класс `FakeAppScope`, который инкапсулировал внутри себя сгенерированные заглушки `MockApiClient` и `MockIFilterStorage`, а также заранее прописывал базовые ответы (возврат пустых списков для `getPlaces` и `getFilter`), чтобы UI мог успешно отрендериться.

Трудность В: Ошибки `Bad state: Cannot call when within a stub response` при тестировании `SettingsModel`.
Решение: В тестах для имитации потоков (`Stream`) первоначально использовался метод `thenReturn`. Согласно правилам `mockito`, для потоков необходимо использовать `thenAnswer((_) => stream)`. Замена метода исправила конфликт внутренних состояний.

Трудность Г: Отсутствие стилей приложения в виджет-тестах. Кнопка `FavoritesButtonWidget` падала из-за невозможности найти `AppColorTheme` в дереве виджетов.
Решение: Тестируемый виджет был обернут в `MaterialApp`, в который принудительно была передана тема приложения `theme: AppThemeData.lightTheme`.

Трудность Д: Ошибки компиляции из-за несоответствия путей к файлам после рефакторинга.
Решение: Были актуализированы пути импортов (например, замена `api/models/` на `api/data/`), а также исправлены имена аргументов в тестовых сущностях (замена `nameFilter` на `name`).
